# Phase 1: Flake Foundation - Research

**Researched:** 2026-04-14
**Domain:** NixOS Flakes, Hyprland, Stylix, home-manager
**Confidence:** HIGH

## Summary

Phase 1 establishes the NixOS flake that boots to a working Hyprland desktop with Stylix theming and home-manager user configuration. This is a well-trodden path in the NixOS ecosystem with mature tooling. The key risks are: incorrect `follows` declarations causing multiple nixpkgs instances, infinite recursion from module import ordering, and GPU-specific Hyprland failures.

The Hyprland module is upstreamed into nixpkgs -- using `programs.hyprland.enable = true` from nixpkgs is now the recommended approach (avoids compiling from source). Stylix lives at `github:nix-community/stylix` and auto-integrates with home-manager when both are NixOS modules. The nerdfonts package has been split -- use `pkgs.nerd-fonts.jetbrains-mono` not the old override syntax.

**Primary recommendation:** Use nixpkgs Hyprland (not the flake), Stylix from nix-community, and home-manager as a NixOS module. Keep the flake inputs minimal: nixpkgs, home-manager, stylix. No Hyprland flake input needed.

## Standard Stack

### Core

| Library | Input URL | Purpose | Why Standard |
|---------|-----------|---------|--------------|
| NixOS (unstable) | `github:NixOS/nixpkgs/nixos-unstable` | Base system, Hyprland package | Required for latest Hyprland; rolling release tracks compositor updates |
| home-manager | `github:nix-community/home-manager` | User-level dotfile management | De facto standard for NixOS user config; master branch tracks nixos-unstable |
| Stylix | `github:nix-community/stylix` | Base16 theming across all components | Only comprehensive theming framework for NixOS; drives colors+fonts from single palette |

### Supporting (from nixpkgs, no extra flake inputs)

| Package | Attribute | Purpose |
|---------|-----------|---------|
| Hyprland | `programs.hyprland.enable` | Wayland compositor (upstreamed NixOS module) |
| Kitty | `pkgs.kitty` | Terminal emulator (Stylix-compatible) |
| JetBrains Mono Nerd Font | `pkgs.nerd-fonts.jetbrains-mono` | Monospace terminal font |
| Noto Color Emoji | `pkgs.noto-fonts-color-emoji` | Emoji font |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| nixpkgs Hyprland | Hyprland flake (`github:hyprwm/Hyprland`) | Flake gives bleeding-edge but requires compilation + cachix setup; nixpkgs version is 1-2 weeks behind but binary-cached |
| Stylix | Manual base16 + per-app config | Stylix handles 30+ apps automatically; manual approach is months of work |
| home-manager NixOS module | Standalone home-manager | NixOS module deploys on `nixos-rebuild switch`; standalone requires separate `home-manager switch` command |

### Why No Hyprland Flake Input

As of April 2026, the Hyprland NixOS module is upstreamed into nixpkgs. Using `programs.hyprland.enable = true` from nixpkgs is the recommended approach for most users. Benefits:
- Binary cached (no compilation)
- No extra flake input to manage
- No `follows` complexity for Hyprland's dependencies
- Version is recent enough on nixos-unstable

Only add the Hyprland flake if a specific bleeding-edge feature is needed later.

## Architecture Patterns

### Recommended Project Structure

```
flake.nix                    # Inputs + nixosConfigurations output
flake.lock                   # Pinned dependencies (auto-generated)
hosts/
  nervos/
    default.nix              # nixosSystem hardware + imports
    hardware-configuration.nix  # Generated by nixos-generate-config
modules/
  nixos/
    hyprland.nix             # programs.hyprland + session config
    gpu.nix                  # GPU-agnostic graphics config
  home/
    default.nix              # home-manager user entry point
    hyprland.nix             # wayland.windowManager.hyprland.settings
    stylix.nix               # Stylix palette, fonts, wallpaper
```

### Pattern 1: Flake Structure with Follows

**What:** All flake inputs use `follows = "nixpkgs"` to share a single nixpkgs instance.
**When to use:** Always -- prevents incompatible library versions and closure bloat.

```nix
# flake.nix
{
  description = "NervOS - Evangelion-themed NixOS";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";

    home-manager = {
      url = "github:nix-community/home-manager";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    stylix = {
      url = "github:nix-community/stylix";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, home-manager, stylix, ... }: {
    nixosConfigurations.nervos = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      specialArgs = { inherit self; };
      modules = [
        ./hosts/nervos
        stylix.nixosModules.stylix
        home-manager.nixosModules.home-manager
        {
          home-manager.useGlobalPkgs = true;
          home-manager.useUserPackages = true;
          home-manager.users.pilot = import ./modules/home;
        }
      ];
    };
  };
}
```

### Pattern 2: Pass Inputs via specialArgs (Not _module.args)

**What:** Use `specialArgs` to pass flake inputs to NixOS modules.
**When to use:** Whenever modules need access to flake inputs.

```nix
# CORRECT: specialArgs is evaluated before module fixpoint
nixpkgs.lib.nixosSystem {
  specialArgs = { inherit self; };
  modules = [ ... ];
};

# WRONG: _module.args causes infinite recursion when used in imports
modules = [{
  _module.args = { inherit inputs; };  # DO NOT DO THIS for imports
}];
```

### Pattern 3: home-manager as NixOS Module

**What:** home-manager integrates as a NixOS module, deploying user config on `nixos-rebuild switch`.
**When to use:** Always in NervOS -- single rebuild command manages everything.

```nix
# In flake.nix modules list:
home-manager.nixosModules.home-manager
{
  home-manager.useGlobalPkgs = true;       # Use system nixpkgs
  home-manager.useUserPackages = true;     # Install to /etc/profiles
  home-manager.users.pilot = import ./modules/home;
}
```

The module export name is `home-manager.nixosModules.home-manager` (also aliased as `home-manager.nixosModules.default`).

### Pattern 4: Stylix Auto-Integration with home-manager

**What:** When Stylix NixOS module detects home-manager, it auto-imports its home-manager module.
**When to use:** Default behavior -- do NOT manually import `stylix.homeModules.stylix` when using both as NixOS modules.

```nix
# CORRECT: Only import the NixOS module; HM integration is automatic
modules = [
  stylix.nixosModules.stylix
  home-manager.nixosModules.home-manager
];

# WRONG: Importing both causes double-definition errors
modules = [
  stylix.nixosModules.stylix
  home-manager.nixosModules.home-manager
  {
    home-manager.users.pilot = {
      imports = [ stylix.homeModules.stylix ];  # DOUBLE IMPORT
    };
  }
];
```

If you ever need manual control: `stylix.homeManagerIntegration.autoImport = false`.

### Anti-Patterns to Avoid

- **Importing Hyprland flake module alongside nixpkgs module:** Causes conflicts. Use one or the other. For NervOS, use nixpkgs.
- **Using `_module.args` for values needed in `imports`:** Causes infinite recursion. Use `specialArgs`.
- **Multiple nixpkgs instances:** Every flake input that doesn't `follows = "nixpkgs"` pulls its own nixpkgs. This bloats closures and causes library mismatches.
- **Manually importing Stylix HM module when using NixOS module:** Causes double-definition errors. Stylix auto-imports into HM.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Theming 30+ apps | Per-app color configs | Stylix base16 system | One palette definition themes everything; manual approach takes months |
| Font management | Manual fontconfig | Stylix fonts declaration | Stylix sets fonts for GTK, terminal, desktop simultaneously |
| User dotfile management | Manual symlinks | home-manager | Declarative, rollback-safe, integrates with NixOS rebuild |
| Session management | Custom login scripts | greetd + Hyprland session | NixOS module handles PAM, environment, session registration |
| GPU driver setup | Manual kernel module loading | `hardware.graphics.enable` | NixOS module handles Mesa, Vulkan, VA-API automatically |

## Common Pitfalls

### Pitfall 1: Infinite Recursion on Module Import

**What goes wrong:** `nix build` fails with "infinite recursion encountered" error.
**Why it happens:** Using `config` values (from `_module.args`) inside `imports = [ ... ]` creates a circular dependency -- imports need to be resolved before config is evaluated.
**How to avoid:** Pass all external values (flake inputs, system name) via `specialArgs`, never via `_module.args` or `config`.
**Warning signs:** Any `imports = [ config.something ]` or `imports = [ args.something ]` where args comes from `_module.args`.

### Pitfall 2: Multiple nixpkgs Instances

**What goes wrong:** Runtime errors about incompatible library versions; enormous closures (10GB+).
**Why it happens:** Each flake input without `follows = "nixpkgs"` downloads its own nixpkgs. Stylix, home-manager each bring their own if not configured.
**How to avoid:** Every input that depends on nixpkgs MUST have `inputs.nixpkgs.follows = "nixpkgs"`. Verify with `nix flake metadata` -- should show one nixpkgs source.
**Warning signs:** `flake.lock` contains multiple `nixpkgs` entries; `nix build` downloads unexpectedly large amounts.

### Pitfall 3: Hyprland Black Screen on Boot

**What goes wrong:** System boots but Hyprland shows black screen, no cursor, no response.
**Why it happens:** Missing graphics drivers, wrong session type, or XWayland issues.
**How to avoid:** Enable `hardware.graphics.enable = true` (required for all GPUs). Use `monitor = ,preferred,auto,auto` for auto-detection. Set `xwayland.enable = true` in Hyprland config.
**Warning signs:** `journalctl --user -u hyprland` shows DRM/GPU errors.

### Pitfall 4: NVIDIA-Specific Failures

**What goes wrong:** Hyprland crashes or shows artifacts on NVIDIA hardware.
**Why it happens:** NVIDIA needs specific kernel modesetting and environment variables.
**How to avoid:** Create a GPU module that's conditionally imported. For NVIDIA: `hardware.nvidia.modesetting.enable = true`, `hardware.nvidia.open = true` (for Turing+), `services.xserver.videoDrivers = ["nvidia"]`.
**Warning signs:** System works in VM (virtio GPU) but fails on NVIDIA hardware.

### Pitfall 5: Nerdfonts Package Renamed

**What goes wrong:** Build error: "nerdfonts has been separated into individual font packages."
**Why it happens:** nixpkgs split the monolithic `nerdfonts` package into individual `nerd-fonts.*` packages.
**How to avoid:** Use `pkgs.nerd-fonts.jetbrains-mono` (NOT `pkgs.nerdfonts` or `nerdfonts.override`).
**Warning signs:** Any reference to `pkgs.nerdfonts` or `nerdfonts.override { fonts = [...]; }`.

### Pitfall 6: Stylix Double-Import with home-manager

**What goes wrong:** Build error about options being "defined twice."
**Why it happens:** Stylix NixOS module auto-imports its home-manager module. Manually importing `stylix.homeModules.stylix` in HM config causes duplication.
**How to avoid:** Only import `stylix.nixosModules.stylix` at the NixOS level. Do NOT import Stylix in home-manager `imports`. If manual control needed: `stylix.homeManagerIntegration.autoImport = false`.
**Warning signs:** Error messages mentioning "option defined in multiple locations" for stylix options.

### Pitfall 7: UWSM Confusion

**What goes wrong:** Conflicting session management between UWSM and home-manager systemd integration.
**Why it happens:** `programs.hyprland.withUWSM = true` conflicts with `wayland.windowManager.hyprland.systemd.enable = true` in home-manager.
**How to avoid:** For NervOS Phase 1, do NOT use UWSM. Set `programs.hyprland.withUWSM = false` (or omit it). UWSM adds complexity without benefit for this use case and has mixed community feedback.
**Warning signs:** Errors about "Hyprland was started without start-hyprland" or duplicate systemd user services.

}

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `hardware.opengl.enable` | `hardware.graphics.enable` | NixOS 24.11 | Old name still works as alias but triggers deprecation warning |
| `nerdfonts.override { fonts = [...]; }` | `nerd-fonts.jetbrains-mono` | nixpkgs ~2024 | Old syntax is a hard error; must use individual packages |
| Hyprland flake input | `programs.hyprland.enable` from nixpkgs | Upstreamed ~2024 | No need for separate flake input; binary cached |
| `github:danth/stylix` | `github:nix-community/stylix` | Moved to nix-community | Old URL may redirect but use canonical nix-community URL |
| Manual UWSM setup | `programs.hyprland.withUWSM` | NixOS 24.11 | Built-in option but UWSM has mixed reception; skip for now |

**Deprecated/outdated:**
- `hardware.opengl.enable`: Renamed to `hardware.graphics.enable` in NixOS 24.11
- `pkgs.nerdfonts`: Split into individual `pkgs.nerd-fonts.*` packages
- UWSM: Described as "more or less deprecated" by some community members; avoid for Phase 1

## Open Questions

1. **Stylix home-manager module name: homeModules vs homeManagerModules**
   - What we know: Official install docs say `stylix.homeModules.stylix`. Some older refs say `stylix.homeManagerModules.stylix`.
   - What's unclear: Whether both work or one is deprecated.
   - Recommendation: Use `stylix.nixosModules.stylix` only (auto-imports HM). This avoids the question entirely.

2. **Display font (Matisse EB) availability in nixpkgs**
   - What we know: Matisse EB is a commercial/rare Japanese font. Unlikely to be in nixpkgs.
   - What's unclear: Whether a close open-source alternative exists.
   - Recommendation: Defer to Phase 3 (Typography). For Phase 1, use DejaVu Sans as placeholder display font. The Stylix font declaration makes swapping trivial later.

3. **VM testing workflow on Windows**
   - What we know: NervOS is developed on Windows, tested in VM first.
   - What's unclear: Whether QEMU/virt-manager or VirtualBox is the intended VM platform.
   - Recommendation: Use QEMU with virtio-gpu for best Wayland support. VirtualBox has poor Wayland/Hyprland support.
