# Phase 2.1: Fork shell - Research

**Researched:** 2026-04-17
**Domain:** Unified Wayland desktop shell (QuickShell/QML) integration into an existing NixOS flake + home-manager + Stylix architecture
**Confidence:** HIGH on shell selection and flake wiring; MEDIUM on Stylix/theming coexistence; LOW on deep QML theming internals (documented elsewhere but not verified here)

## Summary

The architectural pivot replaces the hand-rolled Waybar/Rofi/mako stack (Phase 2 Plans 01-04 and Plan 03-02) with a single upstream shell project vendored in as a flake input. Two candidates were under consideration: **caelestia-dots/shell** and **end-4/dots-hyprland (illogical-impulse)**. Only one has first-party NixOS support worth adopting.

**caelestia-dots/shell** ships its own `flake.nix` with `homeManagerModules.default`, a declarative `programs.caelestia.settings` schema that generates `~/.config/caelestia/shell.json`, a systemd user service bound to `graphical-session.target`, and three build variants (`default`, `with-cli`, `debug`). The shell is QuickShell-based (QML+TS), licensed GPL-3.0, and provides a bar, launcher, dashboard, notifications, lock screen, and sidebar — i.e., everything Phase 2 built by hand plus more.

**end-4/dots-hyprland** has **no flake.nix or nix/ folder in the upstream repository**. NixOS users rely on community third-party flakes (soymou/illogical-flake, celesrenata/end-4-flakes, bigsaltyfishes/end-4-dots) of varying quality and maintenance status. Upstream positions NixOS as a WIP `dist-nix` effort tracked in discussion #2382 and #1093. Adopting end-4 means adopting an unofficial flake and inheriting its maintenance burden.

**Primary recommendation:** Fork **caelestia-dots/shell** into `flake.nix` as a flake input, consume its `homeManagerModules.default` from `modules/home/`, delete the hand-rolled waybar/rofi/mako/lock modules, and keep wallpaper/clipboard/kitty + all Hyprland keybinds. Let Stylix continue to own system-wide typography (Cinzel/Noto Serif Display) and base16, and let caelestia own shell-internal QML theming via its own `settings` schema — do not try to pipe Stylix base16 into caelestia's Material You pipeline; they are architecturally incompatible. Use `config.lib.stylix.colors.base0X` only where caelestia exposes hex overrides in its schema.

### Pattern 1: Flake Input + homeManagerModules.default
**What:** Pull the shell as a flake input, feed its `homeManagerModules.default` into home-manager via `sharedModules`, then configure with `programs.caelestia.settings` in a dedicated home-manager module file.

**When to use:** Always, for any third-party flake that exports a home-manager module. This is the canonical pattern and keeps the flake.nix clean.

**Example:**
```nix
# flake.nix (outputs)
outputs = { self, nixpkgs, home-manager, stylix, caelestia-shell, ... }:
let system = "x86_64-linux"; in {
  nixosConfigurations.nervos = nixpkgs.lib.nixosSystem {
    inherit system;
    specialArgs = { inherit self; };
    modules = [
      ./hosts/nervos
      stylix.nixosModules.stylix
      home-manager.nixosModules.home-manager
      {
        home-manager.useGlobalPkgs = true;
        home-manager.useUserPackages = true;
        home-manager.sharedModules = [ caelestia-shell.homeManagerModules.default ];
        home-manager.users.pilot = import ./modules/home;
      }
    ];
  };
};
```

```nix
# modules/home/caelestia.nix
{ ... }:
{
  programs.caelestia = {
    enable = true;
    systemd = {
      enable = true;                          # user service, starts after compositor
      target = "graphical-session.target";    # canonical Hyprland target
    };
    cli.enable = true;                        # adds `caelestia` to PATH for keybinds
    settings = {
      # Shell config (generates ~/.config/caelestia/shell.json)
      bar.status.showBattery = true;
      paths.wallpaperDir = "~/Pictures/wallpapers";
    };
  };
}
```

Source: caelestia-dots/shell README; DeepWiki Installation & Setup page.

### Pattern 2: Keep Stylix in its Lane
**What:** Let Stylix own system-wide fonts/GTK/Qt/kitty/(non-caelestia) theming; let caelestia own its own QML internals via `programs.caelestia.settings`. Do **not** attempt to inject `config.lib.stylix.colors.base09` into caelestia unless caelestia's schema explicitly accepts hex overrides at that point.

**When to use:** Always. The two systems have overlapping but incompatible color philosophies (base16 vs Material You dynamic palettes).

**Why:** Stylix will theme everything Stylix knows how to theme (kitty, GTK, Qt5ct, cursor). Caelestia's QML shell is NOT a Stylix target, so Stylix won't touch it. Attempting to bridge them produces double-definition errors or requires a Phase 3+ matugen integration that is deferred.

### Pattern 3: Hyprland Keybind Handoff
**What:** Update `modules/home/hyprland.nix` keybinds to call caelestia-cli subcommands instead of rofi/grimblast/cliphist. This preserves the working Phase 2 keybind UX while routing everything through the new shell.

**Example migration:**
```nix
bind = [
  # Was: "$mainMod, SPACE, exec, rofi -show drun"
  "$mainMod, SPACE, exec, caelestia shell drawers toggle launcher"
  # Was: "$mainMod, L, exec, loginctl lock-session"
  "$mainMod, L, exec, loginctl lock-session"   # keep — caelestia lock plugs into systemd
  # Was: cliphist pipeline
  "$mainMod, V, exec, caelestia clipboard"
  # Was: grimblast
  ", Print, exec, caelestia screenshot region"
  "SHIFT, Print, exec, caelestia screenshot screen"
];
```

The exact subcommand names ARE documented in the CLI repo; verify against `caelestia --help` post-install before finalizing.

### Anti-Patterns to Avoid
- **Don't copy upstream dotfiles into the repo.** You want the flake input, not a vendored copy. Vendoring means you lose upstream updates and inherit the maintenance burden the pivot was supposed to escape.
- **Don't manually install quickshell.** caelestia's flake pulls it transitively; installing `pkgs.quickshell` in `environment.systemPackages` creates version skew.
- **Don't hand-write the shell.json.** Use `programs.caelestia.settings` — the home-manager module generates it. Hand-editing `~/.config/caelestia/shell.json` fights home-manager's read-only activation.
- **Don't disable Stylix.** Stylix still themes kitty, GTK apps, cursor, and the system font stack — none of which caelestia manages.
- **Don't import `stylix.homeModules.stylix` anywhere.** Confirmed Phase 01 decision — Stylix auto-integrates via the NixOS module.
- **Don't keep waybar/rofi/mako/lock alongside caelestia.** They will start simultaneously and fight for the same screen real estate. Delete them from `modules/home/default.nix`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Status bar | Waybar config | caelestia shell bar | GTK CSS can't `scaleX(0.8)` (the entire reason for this pivot) |
| App launcher | Rofi drun config | caelestia shell launcher | Already have a fully-themed QML launcher in the same codebase |
| Notifications | mako config | caelestia shell notification service | Single shell lifecycle; no duplicate daemons |
| Lock screen | hyprlock config | caelestia shell lock | Same shell = consistent theming, no separate PAM config beyond `programs.hyprlock.enable` which we already have |
| Wallpaper daemon | hyprpaper (or swww) | caelestia-cli wallpaper | caelestia auto-watches `paths.wallpaperDir`; hyprpaper adds another daemon and another source of truth |
| Clipboard history UI | cliphist + rofi dmenu pipeline | `caelestia clipboard` | One command, one keybind, integrated UI |
| Screenshots | grimblast | `caelestia screenshot` / `caelestia record` | Built-in region/screen/window + annotation via swappy. Also records. |
| Scheme switching daemon | custom Nix script | `caelestia scheme` | Already exists; will integrate cleanly with NixOS specialisations by invoking it from the specialisation's activation script |
| QML widget framework | eww / custom AGS | caelestia's QML Config singleton | Already type-safe, already exposed via shell.json |

**Key insight:** Phase 2 built six modules (waybar, rofi, mako, lock, wallpaper, clipboard plumbing). Phase 2.1 deletes five of them and replaces them with one flake input + one home-manager module file. That is the payoff.

## Common Pitfalls

### Pitfall 1: quickshell Is Not Reliably in Nixpkgs
**What goes wrong:** Attempting to install quickshell via `pkgs.quickshell` outside caelestia's flake fails or provides a stale version.
**Why it happens:** quickshell is packaged via its own flake overlay; nixpkgs coverage has been inconsistent (issue #398 confirms "No native NixOS package or module exists").
**How to avoid:** Let caelestia's flake pull quickshell transitively. Do NOT add `pkgs.quickshell` to `environment.systemPackages`.
**Warning signs:** `error: attribute 'quickshell' missing` at eval time — means you're bypassing the flake.

### Pitfall 2: Stylix Has No Caelestia Target
**What goes wrong:** Expecting Stylix to theme the caelestia bar/launcher the way it themes Waybar.
**Why it happens:** Stylix ships built-in `stylix.targets.waybar`, `stylix.targets.hyprlock`, etc. There is no `stylix.targets.caelestia`. The shell is outside Stylix's target list.
**How to avoid:** Treat caelestia theming as a separate concern. Configure it via `programs.caelestia.settings` (eventually feeding matugen-generated palettes per pilot in a later phase). Stylix still themes kitty, GTK, Qt, cursor — that's enough.
**Warning signs:** Bar colors don't change after editing `stylix.base16Scheme`. This is expected behavior, not a bug.

### Pitfall 3: Two Shells Running Simultaneously
**What goes wrong:** Forgetting to disable `programs.waybar.enable`, `services.mako.enable`, `programs.hyprlock.enable` (home-manager side) produces overlapping bars, duplicate notifications, and broken lock.
**Why it happens:** Home-manager's systemd integration auto-starts all enabled user services.
**How to avoid:** Delete the old module files AND their imports in `modules/home/default.nix`. Use `git rm`, don't just comment out.
**Warning signs:** Two bars visible at top of screen on rebuild.

### Pitfall 4: Hyprland systemd.enable vs. exec-once
**What goes wrong:** Starting caelestia via `exec-once` while the home-manager systemd service is also `enable = true` produces two shell instances.
**Why it happens:** Both init paths fire. The HM module defaults `systemd.enable` to something — check current default before setting.
**How to avoid:** Pick one. Recommended: `programs.caelestia.systemd.enable = true` + `target = "graphical-session.target"`, no exec-once. This matches existing Phase 2 pattern for waybar (`systemd.enable = true`, no exec-once).
**Warning signs:** Duplicate bars/windows; `systemctl --user status caelestia-shell` shows one running, but another was forked by Hyprland.

### Pitfall 5: NixOS Specialisations Must Not Swap Shell Packages
**What goes wrong:** Trying to swap the caelestia flake input per pilot specialisation.
**Why it happens:** Specialisations swap config options, not flake inputs.
**How to avoid:** Keep ONE caelestia shell across all pilots. Change `programs.caelestia.settings` (colors, bar config) per specialisation. This is compatible with how specialisations work: `specialisation.rei.configuration = { programs.caelestia.settings.bar.status.showBattery = false; };`.
**Warning signs:** You find yourself writing `specialisation.rei.inputs.caelestia.url = ...` — stop; this is architecturally wrong.

### Pitfall 6: Font Pivot Not Yet Applied
**What goes wrong:** Phase 3-01 still registers `MatissePro-EB`. Caelestia ships with Rubik + CaskaydiaCove Nerd Font. Until the font pivot is applied, NERV Command typography will regress.
**Why it happens:** Font change is listed in locked decisions but not yet in `modules/nixos/stylix.nix`.
**How to avoid:** Phase 2.1 should NOT do the font swap (that's Phase 3 work). But plan accordingly: acknowledge the shell uses its own fonts and the Cinzel/Noto Serif Display swap lands in a later phase that re-themes caelestia's typography via `programs.caelestia.settings`.
**Warning signs:** None immediately; this is a known gap, not a bug.

### Pitfall 7: License Surface Expansion
**What goes wrong:** Vendoring GPL-3.0 caelestia into a project with mixed licensing.
**Why it happens:** Flake inputs impose license inheritance on downstream distribution (the NervOS ISO, eventually).
**How to avoid:** NervOS is not yet a distributable ISO. When ISO work begins (Phase 10), ensure the full transitive license surface is catalogued. caelestia GPL-3.0 is redistributable — compatible with NervOS-as-a-whole — but any later proprietary component would conflict.
**Warning signs:** None until Phase 10.

## Open Questions

1. **Does caelestia's lock screen reuse the hyprlock PAM service or ship its own?**
   - What we know: Phase 2-02 enabled `programs.hyprlock.enable = true` on the NixOS side for PAM wiring. Caelestia has a "lock screen" component.
   - What's unclear: Whether caelestia's lock is a rebranded hyprlock or a separate QML surface requiring its own PAM service.
   - Recommendation: Keep `programs.hyprlock.enable = true` during Phase 2.1; verify at integration time by running `systemctl --user status` and checking which binary caelestia invokes on `loginctl lock-session`. Revisit in a cleanup pass.

2. **Exact `caelestia` CLI subcommand names for launcher/clipboard/screenshot.**
   - What we know: Subcommands `shell`, `clipboard`, `screenshot`, `record`, `wallpaper`, `scheme`, `emoji`, `toggle` exist (from cli repo).
   - What's unclear: The precise argument grammar for "open the launcher drawer" and "screenshot region vs screen vs window."
   - Recommendation: Plan to run `caelestia --help` and `caelestia shell --help` after first rebuild; finalize keybinds in a follow-up checkpoint within the plan. Do NOT commit keybind strings without verifying.

3. **Should the `with-cli` package variant be used, or `cli.enable = true`?**
   - What we know: The HM module exposes `cli.enable`; the flake also exposes a `with-cli` package.
   - What's unclear: Whether these are equivalent or one is preferred.
   - Recommendation: Use `programs.caelestia.cli.enable = true` in the home-manager module. That is the declarative path and it's documented in the README. Do NOT add the shell to `environment.systemPackages`.

4. **How does Stylix `stylix.targets.hyprlock` interact with caelestia's lock (if caelestia reuses hyprlock)?**
   - What we know: Stylix Phase 3-01 themes hyprlock.
