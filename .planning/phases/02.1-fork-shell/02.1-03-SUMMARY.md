## Resolved Open Questions

### 1. Does caelestia's lock reuse hyprlock?
**No.** caelestia uses `WlSessionLock` (Wayland session lock protocol directly) via its own QML `Lock {}` module at `modules/lock/Lock.qml`. hyprlock is not involved. Evidence: `shell.qml` imports `modules/lock` which contains `WlSessionLock { ... }`.

**Action taken:** `programs.hyprlock.enable = false` in `modules/nixos/hyprland.nix`. Rebuild confirmed: `removing obsolete symlink '/etc/pam.d/hyprlock'`.

### 2. Exact caelestia CLI subcommand grammar

```
caelestia --help output:
  shell        start or message the shell
  toggle       toggle a special workspace
  scheme       manage the colour scheme
  screenshot   take a screenshot
  record       start a screen recording
  clipboard    open clipboard history
  emoji        emoji/glyph utilities
  wallpaper    manage the wallpaper
  resizer      window resizer daemon

caelestia screenshot --help:
  -r, --region [REGION]   take a screenshot of a region (const="slurp" when no value)
  -f, --freeze             freeze the screen while selecting

caelestia shell <message>: sends IPC to running shell via `caelestia-shell ipc call <message>`
```

Launcher is NOT invoked via `caelestia shell ...`. It is a registered `GlobalShortcut { appid: "caelestia"; name: "launcher" }` in `modules/Shortcuts.qml`, triggered via Hyprland's `global` dispatcher.

### 3. programs.caelestia.systemd.enable default
Default is `true` (confirmed from hm-module.nix source). The explicit `systemd.enable = true` in `modules/home/caelestia.nix` is redundant but harmless.

### 4. stylix.targets.hyprlock interaction
`stylix.targets.hyprlock` does not exist in this version of Stylix (nixos-unstable as of 2026-04). The option was removed. No action needed — with `programs.hyprlock.enable = false`, hyprlock is fully absent.
